# Plan to decompose and deliver PBI-020 + PBI-007

## 1) Re-baseline the dependency model
- Keep `PBI-020` independent at backlog level.
- Remove the old cycle permanently:
  - `PBI-020` must **not** depend on `PBI-007`.
  - Only the **activation-related stories** from `PBI-007` should depend on the approval output produced by `PBI-020`.
- Treat `PBI-022` as a **cross-cutting control dependency**, not as a parent delivery chain:
  - approval actions
  - activation actions
  - distribution actions
  must emit audit/non-repudiation events before they can be considered Done.

## 2) Freeze the minimum foundation before PLS implementation starts
Base architecture is considered sufficient to start `PBI-020` and early `PBI-007` work when all of the following are available:
- `PBI-003` completed:
  - role model
  - membership lifecycle
  - access-control primitives
- thin-slice of `PBI-022` completed:
  - event taxonomy for protected actions
  - actor/timestamp/resource logging
  - non-repudiation design for approval and activation events
- baseline domain decisions recorded:
  - PLS contract lifecycle states
  - approval states
  - activation gate rule
  - distribution event shape

## 3) Handle ADRs as support artifacts, not parallel backlog
Create/update these ADRs before implementation stories are pulled into a Sprint:
- `ADR-PLS-001` — PLS contract aggregate and state model
- `ADR-PLS-002` — Shariah approval gate and activation rule
- `ADR-PLS-003` — Audit/non-repudiation events for regulated actions
- `ADR-PLS-004` — Centralized-first execution with optional proof anchoring

ADR handling flow:
1. Draft ADR from refinement/spike output.
2. Review with PO + Developers + relevant compliance/Shariah stakeholder.
3. Mark ADR status as Proposed / Accepted / Superseded.
4. Reference accepted ADR IDs in the child PBIs `Dependencies` or `Notes`.
5. If Sprint Review changes the approach, create a superseding ADR instead of rewriting history.

## 4) Run the uncertainty-reduction spike first
Before implementation stories for `PBI-007` are selected into a Sprint, run one short spike to close the remaining ambiguity:
- approval checklist fields
- approval roles and decision authority
- allowed PLS parameters
- contract activation preconditions
- distribution edge cases requiring explicit rule treatment

Spike output must produce:
- approved field list
- approved workflow states
- rejection / conditional approval handling
- follow-up story confirmation
- ADR updates

## 5) Deliver `PBI-020` first as the governance lane
Sequence:
1. submit review request
2. capture checklist and reviewer inputs
3. record approve / reject / conditional approve decision
4. enforce activation gate from approval state

Goal:
- governance becomes executable before the business workflow is activated

## 6) Deliver `PBI-007` second as the contract lane
Sequence:
1. create contract from approved template/rule set
2. activate contract only if governance gate passes
3. record profit/loss distribution against active contract
4. expose contract lifecycle and audit history

Goal:
- the contract workflow is built on top of already-defined approval semantics

## 7) Development strategy by Sprint wave
### Wave A — Foundations and governance
- finish `PBI-003`
- finish thin slice of `PBI-022`
- run spike
- deliver first `PBI-020` stories
- finalize ADR-PLS-001 to ADR-PLS-003

### Wave B — Contract creation and activation
- deliver contract creation story
- deliver activation gate story
- verify role/access constraints
- verify audit events for create/approve/activate paths

### Wave C — Distribution and traceability
- deliver profit/loss distribution story
- deliver contract history/audit view story
- validate negative cases
- link evidence to backlog and DoD records

## 8) Testing and evidence strategy
For each selected story:
- unit tests for domain rules
- integration tests for approval-to-activation flow
- staging evidence for each Acceptance Criterion
- linked docs/runbook/API update where impacted
- audit evidence attached to the PBI before PO acceptance

## 9) Definition-of-Done enforcement for this slice
A child story is not Done unless:
- approval/activation/distribution behavior works in staging
- audit evidence exists for regulated actions
- docs are updated where behavior or interfaces changed
- no unresolved critical/high vulnerabilities remain for changed components
- PO acceptance comment is recorded
