Title: Adopt centralized evidence pipeline with deterministic hashing, daily manifest, TSA timestamping, and WORM retention
ADR-ID: ADR-2026-002
Status: Accepted

Context:
  - ReqID R05 requires an immutable audit trail for procure-to-pay events.
  - The project is constrained by a centralized-first or hybrid architecture preference, low setup cost, low operational burden, and delivery within 12 weeks.
  - Key stakeholders are the Blockchain Developer / Tech Lead, Product Owner, and Tech Lead.
  - The supporting spike evaluated Fabric private data collections, public-chain anchoring, RFC 3161 timestamping, WORM storage, and QLDB.

Decision:
  - Adopt a centralized evidence pipeline for MVP.
  - The MVP pipeline shall:
    - produce canonical JSON for each audit event
    - hash each event using SHA-256
    - store event hashes in a daily evidence manifest
    - compute a Merkle root for each manifest
    - obtain an RFC 3161 timestamp over the manifest root
    - store the resulting evidence bundle in immutable / WORM storage
    - expose verification logic for auditors
  - Fabric anchoring is deferred as a possible post-MVP enhancement.
  - Expected data flow:
    - serialize each audit event into canonical JSON
    - hash the canonical event using SHA-256
    - append the resulting hash to the daily evidence manifest
    - compute a Merkle root for the manifest
    - obtain an RFC 3161 timestamp over the manifest root
    - store the resulting evidence bundle in immutable / WORM storage
    - verify integrity by recomputing hashes, checking manifest inclusion, and validating the timestamped root

Consequences:
  - Positive consequences (benefits):
    - low implementation and operational cost
    - easy to explain to stakeholders and auditors
    - no blockchain operations required in MVP
    - compatible with stronger anchoring options in later phases
  - Negative consequences (risks, trade-offs):
    - trust model depends partly on the timestamp authority and storage provider
    - does not provide a full distributed-ledger model
    - later Fabric integration may require additional adapter work
  - Migration strategy (if applicable):
    - implement the centralized evidence pipeline as the MVP baseline
    - preserve deterministic bundle formats and verification interfaces so stronger anchoring can be added later
    - add Fabric or other hybrid anchoring in a later phase without replacing the existing evidence-generation flow
